# Clinical Lecture PPT Refinement

## Core Workflow

1. Inspect the working folder first: list `.pptx`, `.md`, index files, source slide folders, and any generation scripts.
2. Read outline/index Markdown before opening large slide libraries. Prefer using the index to choose a small set of relevant source slides.
3. Extract current deck structure with `python-pptx`: slide count, canvas size, titles, picture counts, and obvious text density. See `scripts/audit_slides.py`.
4. **Always create a timestamped backup before bulk edits** (e.g., `<name>_bak_before_<change>.pptx`).
5. If the deck is generated by a script, update the script and regenerate the deck rather than hand-editing the `.pptx` when feasible.
6. Export representative source slides to PNG with PowerPoint COM on Windows when reusing slides visually.
7. Export the revised deck to PNG and **visually verify each modified slide** before finalizing — never trust silent edits.

## Japanese Title Length Rules

Japanese slide titles wrap more unpredictably than English because characters are fixed-width. Apply these rules when generating or reviewing Japanese-language lecture decks:

- **Target: ≤ 20 full-width characters per title line.** Mix of CJK + ASCII is fine; count ASCII runs as roughly half a character each.
- **Preferred pattern**: `キーワード：副題` — lead with the main concept, append detail after a colon. Keep the keyword part (before `：`) to ≤ 10 characters.
- **Always set `MSO_AUTO_SIZE.TEXT_TO_FIT_SHAPE` on title placeholders** as a safety net:
  ```python
  from pptx.enum.text import MSO_AUTO_SIZE
  title_ph.text_frame.auto_size = MSO_AUTO_SIZE.TEXT_TO_FIT_SHAPE
  ```
  This shrinks the font automatically if the title still overflows after shortening.
- **Do not rely on auto-fit alone** — a title that needs to shrink below ~24 pt looks weak on projection. Shorten the text first.
- Avoid quoting brackets (`「」`) in titles; they add 2 characters and a visual hiccup. Move quoted terms to the body text.

## Standard 16:9 Layout System

For a 13.33" × 7.5" widescreen deck, use this consistent layout across content slides so every slide reads the same:

| Element | Position (inches)         | Size            | Style                              |
|---------|--------------------------|-----------------|------------------------------------|
| Accent bar (left of title) | `(0.80, 0.60)` | `0.10 × 0.70`   | Filled solid #29ABE2, no outline   |
| Title text | `(1.10, 0.55)`         | `11.00 × 0.80`  | Meiryo bold sz=2400, color #0F172A |
| Subtitle (optional) | `(1.10, 1.38)`| `11.00 × 0.30`  | Meiryo sz=1400-1500, color #475569 |
| Body content top | `y = 1.55-1.80`  | up to `5.5-5.7` | Body starts after title area       |
| Bottom message (optional) | `y = 6.10-6.30` | full width | Stronger color, ▶ leader          |

When a slide deviates from this (different placeholder layout from a copied source), either accept it as-is (e.g., evidence-image slide with its own caption layout) or rebuild it to match the standard — half-converting creates worse inconsistency than not converting.

## Visual Hierarchy and Color Palette

Medical lecture decks for physicians need clear visual hierarchy. Use this palette consistently across the deck:

| Role | Hex | Use |
|------|-----|-----|
| Title / strong heading | `#0F172A` | Slide title, major dark text |
| Header (■ marker) | `#0284C7` | Section headers in body, bold sz=2000-2400 |
| Body text | `#334155` | Default body, normal weight sz=1600-1900 |
| Quote / patient voice | `#475569` | Italic quotes, attributed speech |
| Subtle / contrast | `#64748B` | "before" half of before/after pairs |
| **Red emphasis** | `#C00000` | **Bold red** for take-home messages, abnormal lab values, key takeaways |
| Warning / amber | `#B45309` | Caveats, ⚠ blocks, secondary cautions |

Patterns:

- **■ blue header → dark slate body** is the default body section structure.
- **▶ red prefix** marks the take-home line for the slide. Keep to one per slide.
- For "patient misconception → doctor's answer" pairs, render the patient line in gray subtle (#64748B) and the doctor line in **red bold** for contrast.
- For lab values, keep normal values dark slate; show **abnormal values in red bold** inline.
- Caveats and "however..." lines use amber (#B45309), not red, so the red reads as the primary signal.

## Projection Layout Rules

- Force true widescreen when needed: `prs.slide_width = Cm(33.867)` and `prs.slide_height = Cm(19.05)`. Do not assume a template is 16:9; many PowerPoint templates are A4 landscape.
- Keep all shapes within slide bounds. After generation, check `left >= 0`, `top >= 0`, `left + width <= slide_width`, and `top + height <= slide_height`.
- Preserve figure aspect ratios. For 16:9 exported slide images, use width/height pairs such as `21.8cm x 12.2625cm`; never stretch to an arbitrary box.
- Use larger text than a document would need. For projected physician lectures, aim for roughly:
  - Titles: 26-34 pt
  - Card body: 14-18 pt
  - Section headers (■): 20-24 pt
  - Table body: 12-15 pt
  - Footers/citations: 7-9 pt
- Avoid slides that are simultaneously small-text and high-whitespace. If whitespace is large, enlarge the text/figure/cards, increase `space_before` between sections, or split the slide.

## Bottom-Whitespace Rebalancing

After a content edit, recheck the bottom margin. If significant blank space remains (>1.5"), do not leave it — it makes the slide look unfinished on projection.

Options, in order of preference:

1. **Increase `space_before` between sections** (e.g., from 400 → 1200 hundredths of a point). This adds breathing room without changing the message.
2. **Increase `line_spc`** on body paragraphs (e.g., 120000-140000 = 120%-140% line spacing).
3. **Increase body font size** by 1-2 pt for body, 2-4 pt for headers.
4. **Promote a single take-home line** to a larger red sz=2200+ closing message.

Avoid simply enlarging every line to fill space — keep the title:header:body proportions sensible.

## Python + lxml Authoring Helpers

`python-pptx` exposes paragraph-level properties only awkwardly. For real styling control, import `scripts/pptx_helpers.py` and build paragraph XML with `lxml`, then inject it into the text frame.

Use these helpers rather than rewriting XML from scratch:

- `make_para()` — one-run paragraph with explicit font, color, bullets, spacing, and line spacing.
- `make_mixed_para()` — multiple inline runs, useful for red bold emphasis inside normal text.
- `replace_paragraphs()` — replace a text frame's existing paragraphs.
- `add_bar()`, `add_title()`, `add_subtitle()` — standard slide header pieces.
- `replace_placeholder_with_textbox()` — remove placeholder auto-bullets and inherited layout colors.
- `insert_slide_after()` / `delete_slides()` — stable slide ordering edits.

Unit reminders:

- `sz` is in hundredths of a point (`1800` = 18 pt).
- `space_before` is in hundredths of a point (`400` = 4 pt).
- `line_spc` is per-mille (`120000` = 120%).

## Common Pitfalls

### Placeholder auto-bullets

A `SlidePlaceholder` body (from "Title and Content" or similar layouts) auto-prepends `•` to every paragraph from the layout's `<a:buChar>` definition. Plain runs are then rendered as bulleted lines even when you didn't intend it.

Fix options:

1. **Add `<a:buNone/>` to every paragraph** in the new XML (the helpers above do this when `no_bullet=True`).
2. **Replace the placeholder with a regular textbox** at the same position. Cleanest result, especially when you also need to change font and add inline emphasis runs:
   ```python
   from pptx.oxml.ns import qn
   sp = placeholder._element
   sp.getparent().remove(sp)
   new_tb = slide.shapes.add_textbox(Inches(L), Inches(T), Inches(W), Inches(H))
   new_tb.text_frame.word_wrap = True
   replace_paragraphs(new_tb.text_frame, paragraphs)
   ```

### Inherited color from layout

If body text mysteriously renders in blue (or any odd color), the layout/master is forcing it via `defRPr`. Setting `font.color` on individual runs may not override it consistently. The XML-level helpers above (with explicit `<a:solidFill>`) are reliable; `python-pptx`'s `.font.color` setters are not.

### Slide insertion and deletion order

To insert a new slide at a specific position, add it to the end then move the element:

```python
new_slide = prs.slides.add_slide(blank_layout)
xml_slides = prs.slides._sldIdLst
new_el = xml_slides[-1]
xml_slides.remove(new_el)
xml_slides.insert(after_idx_0based + 1, new_el)
```

To delete slides, **iterate in reverse order** to keep indices stable:

```python
from pptx.oxml.ns import qn
for idx_1based in sorted([28, 27, 24, 23, 17, 16, 8, 7], reverse=True):
    idx = idx_1based - 1
    el = prs.slides._sldIdLst[idx]
    rId = el.get(qn('r:id'))
    prs.slides._sldIdLst.remove(el)
    prs.part.drop_rel(rId)
```

### Image insertion: title bars and citation strips

Source slides exported as PNG often include a title bar (top ~80-130 px) and citation strip (bottom ~50-150 px). When reusing the chart only, crop with PIL first so the new slide's own title doesn't double up:

```python
from PIL import Image
img = Image.open(src).convert("RGB")
img.crop((15, 130, 1585, 835)).save(dst)
```

Pixel-test the cut line by sampling row brightness; the title bar is usually a different color than the chart background.

### File locks from PowerPoint COM

PowerPoint COM holds the file open even after `prs.Close()`. If you immediately rename or overwrite the file, you may get `PermissionError [WinError 32]`. Either:
- `app.Quit()` and pause briefly, or
- Save to a different filename and rename only after the COM process exits.

## Medical Lecture Structure

For 20-40 minute physician lectures, prefer:

1. One early conclusion slide with 3 takeaways.
2. A time-boxed agenda.
3. Short narrative slides in the speaker's own voice.
4. Selected evidence slides as images, paired with a right-side "how to interpret this" panel.
5. Practical clinic workflow slides: patient selection, dosing/titration, adverse event management, follow-up, referral threshold.
6. A final take-home slide with 4-5 messages.

Do not overload the deck with every available manufacturer slide. Use only the few figures that carry the argument.

## Evidence and Safety

- For current drug indications, dosing, contraindications, or safety language, verify against primary/official sources such as PMDA, package inserts, society guidelines, or original trial publications.
- Keep a clear distinction between diabetes indications and obesity indications.
- In clinical case material, remove or avoid patient identifiers and production data.
- When reusing manufacturer-provided slides, preserve citations and add speaker interpretation rather than silently modifying the evidence figure.

## Windows PowerPoint Export

Use PowerPoint COM via `win32com.client` (Python) when available for visual export of the active deck:

```python
import win32com.client
PPTX = r"C:\path\to\deck.pptx"
app = win32com.client.Dispatch("PowerPoint.Application")
app.Visible = True  # background mode can fail; keep visible
prs = app.Presentations.Open(PPTX, ReadOnly=True, Untitled=False, WithWindow=False)
for n in [7, 13, 17, 18]:
    prs.Slides(n).Export(fr"C:\tmp\s{n:02d}.png", "PNG", 1920, 1080)
prs.Close()
app.Quit()
```

Or PowerShell when running from a shell context:

```powershell
$out = Join-Path (Get-Location) 'review_png'
New-Item -ItemType Directory -Force -Path $out | Out-Null
$pp = New-Object -ComObject PowerPoint.Application
$pres = $pp.Presentations.Open((Join-Path (Get-Location) 'deck.pptx'), -1, 0, 0)
$pres.Export($out, 'PNG', 1600, 900)
$pres.Close()
$pp.Quit()
```

Then inspect title, dense table, evidence-image, adverse-event, and take-home slides. Fix squashed figures, clipped right edges, tiny tables, and excessive whitespace before finalizing.
